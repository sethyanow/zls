## Context

Distinct feature surfaced by Codex review during zls-pun planning. Currently, `findReferences` on an `@import("foo")` string literal returns nothing — the `referencesHandler` only branches into identifier/field/label logic. Imports are not symbol declarations, so the symbol-reference pipeline is the wrong abstraction for them.

`goto.zig:253` already has a separate `import_string_literal` path for goToDefinition — pattern exists to follow. This task adds the reverse-direction counterpart in the references handler.

**Scope boundary (pinned):** this task implements find-references ONLY when the cursor is on an `@import("...")` string literal. It does NOT try to make `findReferences` work on `b.addModule("foo", ...)` callsites in build.zig; that would require a new position-context and reverse-edge mapping for build.zig module names — separate feature, separate task.

## Design (pinned — do not redesign during SRE)

Per Codex review:
- Add an early `pos_context == .import_string_literal` branch in `src/features/references.zig` `referencesHandler` (currently at line 1485 in Server.zig; delegates to `src/features/references.zig:740`).
- Resolve the selected import via `DocumentStore.uriFromImportStr` (line 2026).
- Walk candidate files (use `gatherWorkspaceReferenceCandidates`'s output, including the new `resolved_imports` from zls-mxw).
- For each candidate, walk the AST and inspect `@import` string-literal callsites.
- For each callsite, resolve via `uriFromImportStr` and compare resolved target URI — NOT literal text. This handles the case where the same target is imported under different strings across different root modules.
- Return `Location`s covering the string literal contents.
- `prepareRename` on an `@import` string MUST refuse — import-path/module-name edits are not symbol rename. (Explicit out-of-scope: no rename support.)

## Requirements

- R-I1: `findReferences` on an `@import("X")` string literal returns all source locations where another file has an `@import` whose resolution matches the target URI.
- R-I2: Resolution comparison is by URI, not by literal text. `@import("algorithms")` and `@import("../src/algorithms.zig")` resolving to the same URI must cross-reference each other.
- R-I3: Supports the full resolution surface: file paths, named build modules, `std`, `builtin`, `root`, build dependencies.
- R-I4: Behaves correctly when build config is unresolved — fallback returns whatever the current resolver yields (including `.none` → empty result set), no crash.
- R-I5: `prepareRename` on an `@import` string literal returns null (no rename). Explicitly tested.

2. **TDD — failing tests first** —

   - Test: same target, different strings — `@import("mod_b")` in `a.zig` and `@import("../b/root.zig")` in `c.zig` both resolve to `b.zig`; `findReferences` on either string finds both.
   - Test: `@import("std")` finds other `@import("std")` occurrences across the candidate set.
   - Test: unresolved build config — returns empty (no resolution), no crash.
   - Test: `prepareRename` on `@import("mod_b")` returns null.

3. **Implementation** —
   - In `src/features/references.zig` `referencesHandler`, add an early branch when `pos_context` is `.import_string_literal` (or the equivalent for `@import`).
   - Extract the import string, resolve via `store.uriFromImportStr(arena, handle, import_str)`. If `.none`, return null or empty.
   - Call `gatherWorkspaceReferenceCandidates` to get candidate URIs (now includes `resolved_imports` after zls-mxw).
   - For each candidate handle, walk the AST for `@import` builtin calls with string-literal args. For each one, call `uriFromImportStr` on the string and compare to the target URI.
   - On match, add a `Location` covering the string-literal token's contents (quotes excluded) at the call site's file.
   - Update `src/features/references.zig` `prepareRenameHandler` analog (or wherever prepareRename dispatches) to return null for `.import_string_literal` context. 

4. **Verify** — `zig build test --summary all`, `zig build check`, `zig fmt --check .`.

## Anti-Patterns

- NO raw string matching on `@import(...)` literal text. Codex's first rejected approach. Misses named modules, `root`, `std`, `builtin`, and any case where different strings resolve to the same target.
- NO synthesizing "module declarations" to reuse the symbol-reference pipeline. Codex's second rejected approach. Contaminates the analyser with fake symbols and creates accidental rename semantics.
- NO rename support. Explicitly out of scope. `prepareRename` must return null.
- NO extending this to `b.addModule("foo", ...)` references in build.zig. Distinct feature, distinct position context, not in this task.
- NO duplicating resolution logic — reuse `uriFromImportStr` for both sides of the comparison.

## Key Considerations

- **Candidate set dependency on zls-mxw:** this task's candidate walk relies on `gatherWorkspaceReferenceCandidates` returning the correct set of files, which requires zls-mxw's `resolved_imports` work. If zls-mxw lands first, this task inherits correct candidate discovery. If scheduled in parallel, this task can be tested against a manually-constructed candidate set, but the end-to-end behavior depends on zls-mxw.
- **Location range:** return the range covering the string literal's content (between the quotes), not the whole `@import(...)` expression. Matches how `goto` returns import-string ranges.
- **Multiple occurrences in one file:** a file may `@import("foo")` in several places. Each occurrence is its own Location.
- **Self-reference:** when the cursor is on an `@import` string, that location itself is in the result set only if `includeDeclaration` is true (per LSP convention).
- **Non-string-literal `@import` calls:** if the `@import` argument isn't a string literal (e.g., a compile-time computed string), skip — current `collectImports` already does this (`if (tree.nodeTag(params[0]) != .string_literal) continue;`).
