## Context

Distinct feature surfaced by Codex review during zls-pun planning. Currently, `findReferences` on an `@import("foo")` string literal returns nothing — the `referencesHandler` only branches into identifier/field/label logic. Imports are not symbol declarations, so the symbol-reference pipeline is the wrong abstraction for them.

`goto.zig:253` already has a separate `import_string_literal` path for goToDefinition — pattern exists to follow. This task adds the reverse-direction counterpart in the references handler.

**Scope boundary (pinned):** this task implements find-references ONLY when the cursor is on an `@import("...")` string literal. It does NOT try to make `findReferences` work on `b.addModule("foo", ...)` callsites in build.zig; that would require a new position-context and reverse-edge mapping for build.zig module names — separate feature, separate task.

## Design (pinned — do not redesign during SRE)

Per Codex review:
- Add an early `pos_context == .import_string_literal` branch in `src/features/references.zig` `referencesHandler` (Server.zig:1485 delegates to references.zig:787; line numbers approximate — handle by symbol name).
- Resolve the selected import via `DocumentStore.uriFromImportStr` (at DocumentStore.zig:2068).
- Walk candidate files (use `gatherWorkspaceReferenceCandidates`'s output at references.zig:330, including the new `resolved_imports` from zls-mxw).
- For each candidate, walk the AST and inspect `@import` string-literal callsites.
- For each callsite, resolve via `uriFromImportStr` and compare resolved target URI — NOT literal text. This handles the case where the same target is imported under different strings across different root modules.
- Return `Location`s covering the string literal contents.
- `prepareRename` on an `@import` string MUST refuse — import-path/module-name edits are not symbol rename. (Explicit out-of-scope: no rename support.)

## SRE Clarifications (added 2026-04-15)

Fresh-session review of pinned design. No redesign — gap-filling only:

1. **`prepareRename` actual dispatch location:** `Server.zig:1468` `prepareRenameHandler` (NOT references.zig — the task's hedge "or wherever" resolves here). Early pos_context check must precede the `offsets.identifierLocFromIndex` call, otherwise identifierLocFromIndex-on-string-quote returns null and the failure mode is indistinguishable from "no symbol here." Explicit `.import_string_literal` → return null is the contract.

2. **Fixture needs `c.zig`:** R-I2's test ("same target, different strings") references `tests/fixtures/module_imports/c.zig` — does not exist in the fixture (only a.zig, b.zig, build.zig present). Decision: add `c.zig` with `const b = @import("b.zig");` (relative file-path import to the same target as `a.zig`'s `@import("mod_b")`). No build.zig change required — file-path imports resolve without BuildConfig. This is test-infrastructure scope, not out-of-scope design change.

3. **`collectImports` filter caveat (DocumentStore.zig:534):** The existing `collectImports` helper filters `if (!std.mem.endsWith(u8, import_string, ".zig")) continue;` — the reference walker MUST NOT copy that filter. Module-name imports (`@import("mod_b")`), `std`, `builtin`, `root` don't end in `.zig`. Walk ALL `@import` string literals and let `uriFromImportStr` return `.none` for anything unresolvable.

4. **`UriFromImportStringResult` is a tagged union:** `.none | .one: Uri | .many: []Uri`. Target resolution and each candidate resolution both span all three. Comparison contract:
   - Build a target URI set from `.one`/`.many` (empty for `.none` → return empty per R-I4).
   - For each candidate `@import` site: call `uriFromImportStr`, iterate its URIs, match if any candidate URI is in the target set.
   - `.many` on either side (e.g., `@import("root")` with multiple root source files) requires set-overlap, not URI equality.

5. **String-literal content loc:** use `pos_context.stringLiteralContentLoc(handle.tree.source)` (mirrors goto.zig:248) to get the quotes-excluded range for both the target import_str and the Location range returned at each match site.

6. **Line number drift (symbols are canonical):** `uriFromImportStr` is at DocumentStore.zig:2068, `referencesHandler` at references.zig:787, `gatherWorkspaceReferenceCandidates` at references.zig:330, `prepareRenameHandler` at Server.zig:1468. Task text said 2026/740/327 — drift from intervening commits, harmless.

## Requirements

- R-I1: `findReferences` on an `@import("X")` string literal returns all source locations where another file has an `@import` whose resolution matches the target URI.
- R-I2: Resolution comparison is by URI, not by literal text. `@import("algorithms")` and `@import("../src/algorithms.zig")` resolving to the same URI must cross-reference each other.
- R-I3: Supports the full resolution surface: file paths, named build modules, `std`, `builtin`, `root`, build dependencies.
- R-I4: Behaves correctly when build config is unresolved — fallback returns whatever the current resolver yields (including `.none` → empty result set), no crash.
- R-I5: `prepareRename` on an `@import` string literal returns null (no rename). Explicitly tested.

## Anti-Patterns

- NO raw string matching on `@import(...)` literal text. Codex's first rejected approach. Misses named modules, `root`, `std`, `builtin`, and any case where different strings resolve to the same target.
- NO synthesizing "module declarations" to reuse the symbol-reference pipeline. Codex's second rejected approach. Contaminates the analyser with fake symbols and creates accidental rename semantics.
- NO rename support. Explicitly out of scope. `prepareRename` must return null.
- NO extending this to `b.addModule("foo", ...)` references in build.zig. Distinct feature, distinct position context, not in this task.
- NO duplicating resolution logic — reuse `uriFromImportStr` for both sides of the comparison.

## Key Considerations

- **Candidate set dependency on zls-mxw:** this task's candidate walk relies on `gatherWorkspaceReferenceCandidates` returning the correct set of files, which requires zls-mxw's `resolved_imports` work. zls-mxw is closed, so candidate discovery is correct.
- **Location range:** return the range covering the string literal's content (between the quotes), not the whole `@import(...)` expression. Matches how `goto` returns import-string ranges.
- **Multiple occurrences in one file:** a file may `@import("foo")` in several places. Each occurrence is its own Location.
- **Self-reference:** when the cursor is on an `@import` string, that location itself is in the result set only if `includeDeclaration` is true (per LSP convention).
- **Non-string-literal `@import` calls:** if the `@import` argument isn't a string literal (e.g., a compile-time computed string), skip — current `collectImports` already does this (`if (tree.nodeTag(params[0]) != .string_literal) continue;`).

## Failure Catalog (Adversarial Planning, 2026-04-15)

Grouped by component, not category. Mitigations are structural — not "add a try/catch."

### Component: `referencesHandler` early branch

**Encoding Boundaries: URI equality comparison**
- Assumption: URIs resolved by different `uriFromImportStr` calls compare bytewise for the same target file.
- Betrayal: Relative-path vs absolute-path resolution, `file:///` vs `file://localhost/`, percent-encoding differences, macOS/Windows case-insensitivity. Arena-allocated URI strings are distinct allocations regardless of content.
- Consequence: Silent false-negatives — two importers of the same file never cross-reference because their resolved URIs differ by one character.
- Mitigation: Use `Uri.ArrayHashMap(void)` (the codebase's canonical URI set — already used by `gatherWorkspaceReferenceCandidates`) for the target set. Let the Uri type's equality contract handle normalization; do NOT roll a custom `std.mem.eql(u8, ...)`. If Uri comparison turns out to be bytewise-naive, that's a separate defect to fix in the Uri layer, not in this handler.

**Input Hostility: empty / degenerate import strings**
- Assumption: Parser-validated `.import_string_literal` context implies a non-empty resolvable string.
- Betrayal: `@import("")` is syntactically valid. Pos_context fires on the empty string literal.
- Consequence: `uriFromImportStr` returns `.none` → target set empty → iterate candidates → every callsite's `.none` intersects with empty set → no matches. Outcome is "empty result, no crash" which IS R-I4. Acceptable, but must be tested.
- Mitigation: Target-set-empty short-circuit at top of handler — return empty before walking candidates. O(0) candidates work.

### Component: Per-candidate AST walk

**Input Hostility: malformed `@import` calls**
- Assumption: Every `@import` builtin_call has `params[0]` as a `.string_literal` node.
- Betrayal: `@import()` (zero params), `@import(foo)` (non-string), `@import("a" ++ "b")` (concat — tag would be `.add` or similar), `@import(42)`. All parse as builtin_call with differing param shapes.
- Consequence: Reading `offsets.tokenToSlice` on a non-string-literal node returns garbage bytes. `uriFromImportStr` on garbage returns `.none` but may log errors.
- Mitigation: Mirror the existing `collectImports` checks at DocumentStore.zig:526-529 verbatim: `if (params.len < 1) continue;` then `if (tree.nodeTag(params[0]) != .string_literal) continue;`. This is a STRUCTURAL guard, not defensive — it's the same pattern the rest of the codebase uses.

**Resource Exhaustion: `@import("std")` on large projects**
- Assumption: Candidate walk is acceptably fast for all target imports.
- Betrayal: Every Zig file imports `std`. `findReferences` on `@import("std")` walks N candidates × M imports × `uriFromImportStr` per match check. On a 500-file project that's 500 × ~5 = 2,500 `uriFromImportStr` calls, each potentially touching the handle lock.
- Consequence: Slow but correct. LSP requests that take > a few seconds feel broken even though they complete.
- Mitigation: `uriFromImportStr` caches into `handle.resolved_imports` — repeat calls on the same (handle, import_str) pair are O(1) after the first. First-call cost is inherent. No speculative optimization — correctness first. If profiling shows hot-spot during implementation, revisit. Document the cost in the handler comment.

### Component: `prepareRenameHandler` early-reject

**Temporal Betrayal: pos_context allocation on hot path**
- Assumption: `Analyser.getPositionContext` is cheap enough to call on every prepareRename request.
- Betrayal: prepareRename fires on keystroke-level timing in some clients (e.g., IntelliJ eagerly probing). getPositionContext allocates.
- Consequence: Per-keystroke allocation overhead for what was previously a trivial function.
- Mitigation: getPositionContext is already invoked in referencesHandler / completions / goto on similar cadences — the cost is well-known and acceptable. Use `arena` instead of `server.allocator` so lifetime matches the request. Do NOT add a "is this pos a string" fast path; that's defensive duplication of getPositionContext's job.

**State Corruption: cursor on identifier adjacent to import**
- Assumption: `.import_string_literal` pos_context fires exactly when cursor is inside `@import("...")` quotes.
- Betrayal: Edge positions — cursor on the `@` builtin token, cursor on the open-paren, cursor between `import` and `(` — may yield different pos_contexts (`.builtin`, `.parens_expr`, etc.). prepareRename behavior at those positions is unchanged from today.
- Consequence: No regression — existing identifierLocFromIndex path naturally handles these.
- Mitigation: Only gate on the EXACT `.import_string_literal` variant. Do not generalize.

### Component: Tests

**Dependency Treachery: R-I3 (`@import("std")`) depends on `zig_lib_dir`**
- Assumption: Test context has `zig_lib_dir` configured so `uriFromImportStr("std")` returns `.one: <lib>/std/std.zig`.
- Betrayal: Test context (`tests/context.zig`) may not set `zig_lib_dir` by default. If `config.zig_lib_dir` is null, `uriFromImportStr("std")` returns `.none` (DocumentStore.zig:2101) — target set empty — test expects cross-references but gets empty.
- Consequence: R-I3 test reports "works correctly: got empty set" even though std handling is actually broken under real config.
- Mitigation: **STRUCTURAL fix**: in R-I3 test setup, explicitly configure `ctx.server.config_manager.config.zig_lib_dir` to a test-owned directory containing a stub `std/std.zig`. Two fixture files (`std/std.zig` placeholder + pointing `zig_lib_dir` at it) prove the URI comparison works, independent of system Zig install. If that's infeasible, test the std path via the live LSP demo in zls-pun and document the R-I3 test limitation — but TRY the fixture approach first.

**Temporal Betrayal: `resolved_imports` cache warming order**
- Assumption: `gatherWorkspaceReferenceCandidates` finds all cross-file importers without prior cache priming.
- Betrayal: `resolved_imports` only populates on `uriFromImportStr` calls. If module-name imports haven't been resolved yet when the reference search runs, candidate discovery misses them.
- Consequence: Test flakiness — same test passes after `goto` triggers the cache, fails in isolation.
- Mitigation: Follow the mxw test pattern at `tests/lsp_features/references.zig:1313-1317` — explicitly call `uriFromImportStr` on module-name imports during test setup to warm the cache. This is a STRUCTURAL part of test setup, not a workaround.

**State Corruption: `c.zig` fixture collision with existing mxw tests**
- Assumption: Adding `c.zig` to `tests/fixtures/module_imports/` doesn't affect the zls-mxw tests using the same fixture.
- Betrayal: mxw tests `didOpen` a.zig + b.zig and assume those are the only files. If they iterate the fixture directory or if `c.zig`'s import of `b.zig` changes the reverse-candidate set by side effect… wait, no — mxw tests didOpen explicitly by file list, not by scanning. Adding c.zig doesn't change their behavior.
- Consequence: None (verified by reviewing mxw test code structure).
- Mitigation: Confirm during TDD by running the existing mxw tests first, before any code changes, to capture the green baseline. If they break after adding c.zig, STOP and investigate.
