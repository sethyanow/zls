## Context

Task 1 (zls-t17) shipped `prepareCallHierarchy` ŌĆö given a position, it returns a `CallHierarchyItem` describing the callable with `data = { "uri": string, "node": integer }` encoded as `std.json.Value.object`.

Task 2 implements the second step of the LSP call hierarchy protocol: given an `Item`, return the `IncomingCall[]` describing every caller. The client invokes `callHierarchy/incomingCalls` with an Item payload; the server must:

1. Decode `Item.data` to recover `(uri, node_index)` ŌĆö the callable's identity.
2. Resolve the (handle, node) pair back to an `Analyser.DeclWithHandle` so existing reference-gathering infrastructure can be reused.
3. Find every call site that targets this decl across the workspace ŌĆö including files transitively reachable via `@import` (Phase 1's eager loading guarantees those files are indexed).
4. For each call site, determine the enclosing caller function via `Analyser.innermostScopeAtIndexWithTag(..., .initOne(.function))`.
5. Group call sites by `(caller_handle, caller_fn_node)` ŌĆö each unique caller collapses to a single `IncomingCall` whose `fromRanges` lists every location where the caller invokes the target.
6. Build a `CallHierarchyItem` for each caller (re-using Task 1's `buildItemIfCallable`).

## Design

### R8 interpretation ŌĆö "Builder extension"
The epic's R8 requires "Builder extension for call hierarchy, consumable by both findReferences and call hierarchy (Approach C ŌĆö unified codepath)." Two Builders exist: the main `Builder` (line 63, general references) and `CallBuilder` (line 556, calls-only). The pragmatic interpretation: **extend `CallBuilder` in place to also record each call site's enclosing function node**. Main Builder stays untouched ŌĆö findReferences doesn't need call context. call_hierarchy.zig consumes the enriched CallBuilder output.

This preserves the "unified codepath" spirit (CallBuilder remains the single call-gathering implementation) without forcing a broader rewrite that Task 2's scope doesn't justify. If a deeper unification of Builder + CallBuilder is warranted later, it can be its own task ŌĆö the epic doesn't mandate collapsing them into one struct.

### Decoding Item.data
Decoder lives in `src/features/call_hierarchy.zig` as a private helper:
```
fn decodeItemData(data: ?types.LSPAny) !?struct {
    uri_raw: []const u8,
    node: Ast.Node.Index,
}
```
Returns null when `data` is null, or when the object is missing `uri` / `node` fields, or when `node.integer` exceeds `tree.nodes.len`. These are "malformed item" conditions ŌĆö graceful null return per LSP etiquette (the client can re-prepare).

Staleness validation: after decoding, resolve the handle via `server.document_store.getHandle(uri)`. Check that `node < handle.tree.nodes.len` and that `tree.nodeTag(node)` is still a callable tag (fn_decl / fn_proto* / test_decl / @"comptime"). If the tree has been re-parsed since prepare, the node_index may point at an unrelated construct ŌĆö validate before using.

### Resolving (handle, node) ŌåÆ DeclWithHandle
For callable nodes, the node itself IS the decl's AST node. Construct directly:
```
const decl_handle: Analyser.DeclWithHandle = .{
    .decl = .{ .ast_node = node },
    .handle = handle,
};
```
Confirm the pattern against `callsiteReferences`'s `decl_handle.decl == .ast_node` assertion at line 641.

### Grouping call sites by caller
After `callsiteReferences` returns `[]NodeWithHandle`, iterate and bucket by (caller_handle, caller_fn_node):
- For each call site `(handle, call_node)`:
  - Compute `source_index` = `offsets.nodeToLoc(&handle.tree, call_node).start`
  - Get `doc_scope` for the handle (via `DocumentStore.Handle.getDocumentScope(allocator)` ŌĆö caches internally)
  - Find enclosing fn: `Analyser.innermostScopeAtIndexWithTag(doc_scope, source_index, .initOne(.function))` ŌåÆ scope (already cast to the scope index type; unwrap is `.unwrap().?` if position is inside any scope).
  - From the scope, derive the fn's AST node. The scope struct carries a `.ast_node` (scope-to-node mapping is established elsewhere in ZLS ŌĆö verify API during implementation).
  - Key: `(handle.uri.raw, caller_fn_node_as_usize)` ŌĆö use `std.HashMap` with this composite key, or a simple `std.ArrayList(Bucket)` with linear scan (few callers per document is the realistic case).
- For each bucket: `fromRanges` is the list of call_node ranges; `from` is a CallHierarchyItem constructed via Task 1's `buildItemIfCallable` applied to (caller_handle, caller_fn_node).

### Handling test_decl and comptime
Tests and comptime blocks cannot be called by user code. incomingCalls on them returns an empty slice (`&.{}`), not null. Null is "method not applicable"; empty is "applicable but no results."

## Anti-Patterns (FORBIDDEN for this task)

- **Creating a second CallBuilder-equivalent instead of extending the existing one.** R8 mandates a unified codepath. Either extend CallBuilder in place OR document a concrete blocker and escalate.
- **Re-resolving the callable from position instead of decoding Item.data.** Task 1 encoded {uri, node} specifically so this task doesn't walk the AST again. If the decoder looks at `item.range.start`, that's the anti-pattern.
- **Returning null for test_decl / comptime Items.** They are valid Items with no callers; return `&.{}`, not null.
- **Advertising `callHierarchyProvider` capability.** Still deferred until all three handlers (incoming + outgoing) ship. Task 3 flips the flag.
- **Skipping the cross-file test.** R2 is explicit: "spanning all files reachable via the import graph." Single-file-only passes prove nothing ŌĆö the whole point of Phase 1's loader is to enable this.

## Key Considerations

### Scope ŌåÆ AST node derivation
`innermostScopeAtIndexWithTag` returns a `Scope` or scope index. The mapping from scope ŌåÆ its defining AST node is: scopes of `.function` tag are created for fn_decls and wrap their AST node. Verify during implementation: `scope.data.ast_node` or `doc_scope.scopes.items[scope_index].ast_node`. If the API differs, fall back to: walk the enclosing nodes via `ast.nodesOverlappingIndexIncludingParseErrors` and find the first `.fn_decl` ancestor ŌĆö same semantics as Task 1's walker, applied to the call's source_index.

### Top-level calls (no enclosing fn)
A call at file scope (e.g., inside a `comptime { target(); }` block that isn't inside any fn) has no caller function. Grouping key becomes `(handle.uri, null)`. Either:
- Emit an IncomingCall with `from` = a synthetic Item representing "file scope" (problematic ŌĆö what Item name?).
- Skip these calls entirely (simpler ŌĆö most LSP clients don't need top-level call tracking for a call hierarchy view).
**Decision:** Skip top-level calls. Document in implementation comment. If user feedback later requests coverage, it's additive.

### Decoder allocation
The decoder runs on the request arena. It returns a struct of borrowed slices (URI points into the JSON string owned by the arena). Safe for the request's lifetime.

### Scale: number of callers
Linear-scan grouping is O(N┬▓) on calls, O(N) on callers. For realistic codebases a function has < 100 callers ŌĆö linear is fine. If profiling later shows this as a hot path, switch to HashMap keyed on `(uri_ptr, node.int)`. Don't optimize preemptively.

### Fixture strategy for cross-file
If `addDocument` with `untitled://` can't simulate `@import` resolution, use real files committed under `tests/fixtures/call_hierarchy/`. Check existing test patterns ŌĆö `tests/fixtures/eager_load/` was used in Phase 1's testing and provides a template.
