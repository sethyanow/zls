## Context

Task 1 (zls-t17) shipped `prepareCallHierarchy` — given a position, it returns a `CallHierarchyItem` describing the callable with `data = { "uri": string, "node": integer }` encoded as `std.json.Value.object`.

Task 2 implements the second step of the LSP call hierarchy protocol: given an `Item`, return the `IncomingCall[]` describing every caller. The client invokes `callHierarchy/incomingCalls` with an Item payload; the server must:

1. Decode `Item.data` to recover `(uri, node_index)` — the callable's identity.
2. Resolve the (handle, node) pair back to an `Analyser.DeclWithHandle` so existing reference-gathering infrastructure can be reused.
3. Find every call site that targets this decl across the workspace — including files transitively reachable via `@import` (Phase 1's eager loading guarantees those files are indexed).
4. For each call site, determine the enclosing caller function via `Analyser.innermostScopeAtIndexWithTag(..., .initOne(.function))`.
5. Group call sites by `(caller_handle, caller_fn_node)` — each unique caller collapses to a single `IncomingCall` whose `fromRanges` lists every location where the caller invokes the target.
6. Build a `CallHierarchyItem` for each caller (re-using Task 1's `buildItemIfCallable`).

## Design

### R8 interpretation — "Builder extension"
The epic's R8 requires "Builder extension for call hierarchy, consumable by both findReferences and call hierarchy (Approach C — unified codepath)." Two Builders exist: the main `Builder` (line 63, general references) and `CallBuilder` (line 556, calls-only). The pragmatic interpretation: **extend `CallBuilder` in place to also record each call site's enclosing function node**. Main Builder stays untouched — findReferences doesn't need call context. call_hierarchy.zig consumes the enriched CallBuilder output.

This preserves the "unified codepath" spirit (CallBuilder remains the single call-gathering implementation) without forcing a broader rewrite that Task 2's scope doesn't justify. If a deeper unification of Builder + CallBuilder is warranted later, it can be its own task — the epic doesn't mandate collapsing them into one struct.

### Decoding Item.data
Decoder lives in `src/features/call_hierarchy.zig` as a private helper:
```
fn decodeItemData(data: ?types.LSPAny) !?struct {
    uri_raw: []const u8,
    node: Ast.Node.Index,
}
```
Returns null when `data` is null, or when the object is missing `uri` / `node` fields, or when `node.integer` exceeds `tree.nodes.len`. These are "malformed item" conditions — graceful null return per LSP etiquette (the client can re-prepare).

Staleness validation: after decoding, resolve the handle via `server.document_store.getHandle(uri)`. Check that `node < handle.tree.nodes.len` and that `tree.nodeTag(node)` is still a callable tag (fn_decl / fn_proto* / test_decl / @"comptime"). If the tree has been re-parsed since prepare, the node_index may point at an unrelated construct — validate before using.

### Resolving (handle, node) → DeclWithHandle
For callable nodes, the node itself IS the decl's AST node. Construct directly:
```
const decl_handle: Analyser.DeclWithHandle = .{
    .decl = .{ .ast_node = node },
    .handle = handle,
};
```
Confirm the pattern against `callsiteReferences`'s `decl_handle.decl == .ast_node` assertion at line 641.

### Grouping call sites by caller
After the extended `callsiteReferencesWithCaller` returns `[]CallSite`, iterate and bucket by (caller_handle, caller_fn_node):
- For each CallSite `{ handle, call_node, caller_fn_node }`:
  - If `caller_fn_node == null`, the call is at file scope (outside any fn) — **skip it** (see Top-level calls decision in Key Considerations).
  - Key: `(handle.uri.raw, caller_fn_node.?)` — use a simple `std.ArrayList(Bucket)` with linear scan (few callers per document is the realistic case).
- For each bucket: `fromRanges` is the list of `offsets.nodeToRange(&handle.tree, call_node, encoding)` for every call_node in the bucket; `from` is a CallHierarchyItem constructed via Task 1's `buildItemIfCallable(arena, handle.uri, &handle.tree, caller_fn_node, null, encoding)` (parent_tag=null — the caller_fn_node is already a fn_decl because scope-of-tag-function wraps fn_decls, not fn_proto children).

The CallBuilder extension (Step 5) is responsible for computing `caller_fn_node` from each call's source_index using `innermostScopeAtIndexWithTag` + `doc_scope.getScopeAstNode(...)`. The handler (Step 6) consumes the precomputed value — it does not re-run scope analysis.

### Handling test_decl and comptime
Tests and comptime blocks cannot be called by user code. incomingCalls on them returns an empty slice (`&.{}`), not null. Null is "method not applicable"; empty is "applicable but no results."

## Key Considerations

### Scope → AST node derivation (RESOLVED)
Locked API: `doc_scope.getScopeAstNode(scope_index).?` — defined at `src/DocumentScope.zig:1279`, used at 6 call sites across ZLS. Returns `?Ast.Node.Index`; the `.?` unwrap is safe for `.container` / `.function` / `.block` scope tags (returns null only for `.other`). Since we filter with `.initOne(.function)`, the unwrap is always safe.

### Top-level calls (no enclosing fn)
A call at file scope (e.g., inside a `comptime { target(); }` block that isn't inside any fn) has no caller function scope — `innermostScopeAtIndexWithTag(.initOne(.function))` returns `.none`. Grouping key would be `(handle.uri, null)`. Two options:
- Emit an IncomingCall with `from` = a synthetic Item representing "file scope" (problematic — what Item name?).
- Skip these calls entirely (simpler — most LSP clients don't need top-level call tracking for a call hierarchy view).
**Decision:** Skip top-level calls. The CallBuilder still records them with `caller_fn_node = null` (cheap, preserves information); the handler filters them out during grouping. Documented in Step 6 flow.

**Known limitation flagged:** Task 1's `prepareCallHierarchy` treats a top-level `comptime { target(); }` block as a **callable** Item (it has a call, so it's recognized as a caller construct). But Task 2's incomingCalls skips the call from that block because the scope-tag filter only recognizes `.function` scopes, not comptime blocks (which are `.block` scope tag). This is a consistency gap — a comptime block can be *prepared* as a caller Item by Task 1 but does not appear as a caller of its callees in Task 2's output. Acceptable for Task 2 (matches "skip top-level" decision); future work could add `.block`-scope-with-comptime-ancestor detection as a caller-finding rule.

### Decoder allocation
The decoder runs on the request arena. It returns a struct of borrowed slices (URI points into the JSON string owned by the arena). Safe for the request's lifetime.

### Scale: number of callers
Linear-scan grouping is O(N²) on calls, O(N) on callers. For realistic codebases a function has < 100 callers — linear is fine. If profiling later shows this as a hot path, switch to HashMap keyed on `(uri_ptr, node.int)`. Don't optimize preemptively.

### Adversarial Failure Catalog

Findings below are grouped by component. Skipped categories are noted per-component with reasoning.

#### decodeItemData (Step 4)

**Input Hostility: Shape mismatch**
- Assumption: `data.object.get("uri")` is a `.string`; `data.object.get("node")` is an `.integer`.
- Betrayal: Hostile client constructs Item with `uri = .{.integer = 42}` or `node = .{.string = "0"}`. Or JSON parses into unexpected variant.
- Consequence: Naive `.?.string` / `.?.integer` causes unreachable panic (invalid enum access) in debug; UB in release.
- Mitigation: Decoder uses `if (value == .string) |s|` pattern for each field before accessing. Return null on any variant mismatch. Implement as explicit switch, not unwrap.

**Input Hostility: Extreme integer values for node**
- Assumption: node.integer fits in `u32` and represents a valid `Ast.Node.Index`.
- Betrayal: Client sends `node = .{.integer = -1}` or `node = .{.integer = 9999999999}` (overflow u32) or `node = .{.integer = 0}` (valid enum but may point at tree root).
- Consequence: `@intCast(u32, node)` with overflow = Zig runtime safety panic in debug. Root node (index 0) may trigger unexpected callable match if root-tag is something rare.
- Mitigation: Decoder checks `node.integer >= 0 and node.integer < tree.nodes.len` BEFORE `@intCast`. Use `std.math.cast(u32, node.integer)` which returns null on overflow rather than panic.

**Input Hostility: Bogus but well-typed uri**
- Assumption: URI string parses as a valid URI.
- Betrayal: `uri = ""` (empty) or `uri = "not-a-uri"` or `uri = "file:///∅/🙈.zig"` (unicode).
- Consequence: `Uri.parse` returns error; we propagate as `InvalidParams` or return null.
- Mitigation: Handler catches `Uri.parse` error and returns null (per Step 6 flow item 2). Unicode paths are valid — `Uri.parse` handles percent-decoding.

**Temporal Betrayal: Tree reparsed between prepare and incomingCalls**
- Assumption: decoded.node still refers to the same callable.
- Betrayal: didChange arrived; tree reparsed; nodes renumbered. node index may now point at a non-callable, out-of-bounds, or semantically different node.
- Consequence: Wrong function's callers returned, or panic on out-of-bounds access.
- Mitigation: Step 6 flow item (3) — bounds check + tag check (callable vs non-callable) BEFORE constructing DeclWithHandle. Covered.

**Encoding Boundaries / Dependency Treachery / State Corruption / Resource Exhaustion:** Skipped — pure function, no external calls, no persistent state, bounded output size.

#### CallBuilder extension (Step 5)

**Input Hostility: Empty tree / edge-case source**
- Assumption: Every call_node has a non-root source_index.
- Betrayal: Call_node IS the root (extremely unlikely given only .call* tags match). `offsets.nodeToLoc(tree, .root).start = 0` — scope iterator from position 0 returns root container scope.
- Consequence: `innermostScopeAtIndexWithTag(.initOne(.function))` on position 0 returns `.none` (no function scope at file scope). caller_fn_node = null. Not dangerous.
- Mitigation: Structural — nullable caller_fn_node handles this case uniformly.

**Dependency Treachery: handle.getDocumentScope fails**
- Assumption: `handle.getDocumentScope()` succeeds.
- Betrayal: Severe parse error, OutOfMemory during scope build.
- Consequence: Propagates as Analyser.Error up the stack. Handler returns error to client.
- Mitigation: Existing behavior — Analyser.Error already includes all scope-build failure modes. No new mitigation needed.

**Resource Exhaustion: Scope lookups per call site**
- Assumption: Caller count is bounded (< 1000 typical).
- Betrayal: Target is a frequently-used function (`std.mem.eql`, `std.fmt.format`). 10,000+ callers across workspace.
- Consequence: O(callers × scope_depth) computations. Seconds-long incomingCalls response. May timeout client.
- Mitigation: Task 2 accepts this cost — scope_depth is typically < 10. LSP's `partialResultToken` field is present in IncomingCallsParams but NOT implemented here (deferred). Document as known behavior; performance work is a future task if profiling shows regression.

**Temporal Betrayal: Tree mutation during walk**
- Assumption: Each handle's tree is stable for request lifetime.
- Betrayal: Concurrent thread modifies tree mid-walk (e.g., background didChange reprocessing).
- Consequence: Read-after-free or wrong data.
- Mitigation: Inherit DocumentStore's existing concurrency invariant — handles are immutable for request duration. No new safety work required; this risk is shared across all ZLS features.

**Encoding Boundaries / State Corruption:** Skipped — internal byte offsets, request-local arena-allocated state.

#### callsiteReferencesWithCaller + thin wrapper (Step 5)

**Input Hostility: Thin wrapper preserves semantics**
- Assumption: Existing consumer at `src/analysis.zig:1775` iterates `refs.items` treating each element as `NodeWithHandle { node, handle }`.
- Betrayal: The projection drops top-level calls (where caller_fn_node is null) — but the existing consumer doesn't care about caller, only call_node + handle.
- Consequence: If we filter out top-level calls during projection, analysis.zig:1775 sees fewer refs than before — regression in callsite-based type inference.
- Mitigation: **Thin wrapper MUST NOT filter.** Project ALL CallSites to NodeWithHandle regardless of caller_fn_node nullity. The handler (Step 6) does the filtering, not the wrapper.

#### incomingCallsHandler (Step 6)

**Temporal Betrayal: Caller file edited between prepare and incomingCalls**
- Assumption: Caller ranges remain valid.
- Betrayal: User edits caller.zig after prepare, before incomingCalls. callsiteReferences re-walks CURRENT trees, so ranges point at current positions — not prepare-time positions.
- Consequence: This is actually CORRECT behavior — LSP wants live data. If the caller was deleted, it's not in results (CallBuilder finds it or doesn't based on current state).
- Mitigation: Structural — CallBuilder always walks fresh trees. No mitigation needed; call this out in code comment to prevent future "caching" optimization that would break correctness.

**Resource Exhaustion: Huge response size**
- Assumption: IncomingCall list fits in a reasonable JSON response (< few MB).
- Betrayal: Target with 10k callers produces a response with 10k IncomingCall entries × K ranges per entry.
- Consequence: Large LSP message may be slow to serialize/transmit. Client may have its own limits.
- Mitigation: Task 2 does not support `partialResultToken` despite the field being present in IncomingCallsParams. Document this limitation in Out of Scope. Future work if needed.

**State Corruption: Arena lifetime**
- Assumption: All returned data (URIs, Items, Ranges) lives in the request arena.
- Betrayal: Bucket's `ranges: std.ArrayList(types.Range)` uses `arena` allocator; `buildItemIfCallable` uses `arena`. All good. But: what if we accidentally use `server.allocator` somewhere? Items would be freed before response serialization.
- Consequence: Use-after-free — corrupt response.
- Mitigation: Structural — pass `arena` consistently. Never touch `server.allocator` in the handler. Code review gate.

#### Server.zig wiring (Step 2)

**Input Hostility: Missing Item fields in request**
- Assumption: lsp_kit parser validates IncomingCallsParams shape.
- Betrayal: Client sends `{"item": null}` or `{"item": {}}` (missing required fields).
- Consequence: lsp_kit returns JSON parse error before reaching handler. Handler never sees malformed params.
- Mitigation: Structural — delegated to lsp_kit. No action needed.

Other categories: Skipped (wiring is purely mechanical dispatch; no independent state, timing, or resource behavior).

#### testIncomingCalls helper (Step 1)

**State Corruption: Cross-test pollution**
- Assumption: Each test creates fresh `Context`.
- Betrayal: If we ever cache Context (for perf), document state could leak across tests.
- Consequence: Test ordering dependencies, flaky failures.
- Mitigation: Mirror existing Task 1 pattern — `var ctx: Context = try .init(); defer ctx.deinit();` per test. No shared state.

Other categories: Skipped — test helpers are not exposed to hostile inputs, don't persist state, don't call external services.

### Fixture strategy for cross-file (RESOLVED — no fixtures needed)
`addDocument` with `untitled://` URIs **does** simulate `@import` resolution. Verified at `tests/lsp_features/references.zig:304-351`:
- `test "cross-file reference"` — two files, `@import("Untitled-0.zig")` resolves correctly.
- `test "cross-file - transitive import"` — three files in a chain.
- `test "cross-file - alias"` — import with field access.

Pattern for Task 2's cross-file test: call `ctx.addDocument(.{ .source = file_a })` first (becomes `untitled:///Untitled-0.zig`), then `ctx.addDocument(.{ .source = "const a = @import(\"Untitled-0.zig\"); fn c() void { a.target(); }" })` (becomes `Untitled-1.zig`). Prepare at `target` in file a, incomingCalls should return a caller in file b.

No file-system fixtures needed. If the test fails, the problem is in Phase 1's eager loader, not the test infrastructure — STOP and surface.
